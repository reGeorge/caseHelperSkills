# dot1x 密码复杂度自动化用例搭建复盘

> 背景：为 66 条 dot1x 密码复杂度手工用例搭建自动化，包括创建用例、公共步骤设计、
> 配置步骤引用替换、以及审计校验全流程。以下按高维和细节两个层面总结。

---

## 一、高维问题（流程 & 方法论）

### 1.1 用例分析阶段："没吃透"手工用例就动手

**问题**: 直接从手工用例标题出发批量建用例，没有先把「标题、前置条件、操作步骤、预期结果」
四个维度的文本拆解为结构化的状态组合。

**后果**: 生成的 66 条用例引用了错误的公共配置步骤——标题说"开启首次+弱密码+定时"，
但实际引用的公共步骤是3个开关全开，或者只是一个笼统的"开启密码管控"，
根本不区分首次/弱密码/定时三个独立维度的排列组合。

**应有流程**:
```
手工用例文本 → 提取所有状态维度 → 枚举全部组合 → 对每个组合确认 API 参数
→ 设计公共步骤 → 再批量创建业务用例
```

**教训**: 在动手建任何用例之前，必须先输出一份「**状态组合 × 公共步骤映射表**」，
明确每种组合对应哪个公共步骤、步骤的 HTTP body 长什么样。
这是整个自动化的骨架，骨架不对，后面全部要返工。

### 1.2 公共步骤设计阶段：颗粒度不够

**问题**: 最初只创建了 3 个公共配置步骤（全开 66888、关首次 66889、关弱密码 66890），
加上一个关闭步骤 66891，一共 4 个。但 3 个开关的排列组合是 2³ = 8 种。

**后果**: 37/80 条业务用例找不到匹配的公共步骤，引用了近似但不正确的配置。
更严重的是，66889 和 66890 的步骤 body 里 dot1x 相关字段全部为 0，
等于它们虽然叫"关闭首次"/"关闭弱密码"，实际上把 dot1x 的所有配置都关了。

**应有流程**:
```
1. 列出所有独立开关维度（首次、弱密码、定时）
2. 枚举 2^n 种组合
3. 对每种组合，从已知正确的 body 模板派生（而非手工拼凑）
4. 建好全部 8 个公共步骤后再开始业务用例
```

**教训**: 公共步骤的颗粒度 = 独立维度的全排列。漏一个组合，就会有一批用例引用错误。

### 1.3 缺少"先验证、再批量"的节奏

**问题**: 拿到 body 模板后直接跑批量脚本，没有先用单条 API 调用验证参数格式。

**后果**: 批量脚本 Phase A 创建了 4 个空壳用例（67211-67214，有用例无步骤），
Phase C 37 条替换全部失败。两次批量执行都因为同一个 `method` 字段类型错误报废。

**应有流程**:
```
1. 先手动调一次 create_step，确认参数格式 → 成功
2. 再手动调一次 update_step，确认更新模式 → 成功
3. 确认 OK 后才启动批量脚本
```

**教训**: 对平台 API 不确定的字段，永远先单条验证。批量操作的代价是 N 倍放大错误。

### 1.4 审计/校验应当前置，而非事后补救

**问题**: 先建了 66 条用例，上线后才开发 case-debugger 做审计，
审计发现 37 条 CONFIG_MISMATCH 后再回头重构公共步骤。

**应有流程**:
```
设计公共步骤(8种) → 建少量样本用例(每种组合1条) → 跑审计 → 全部通过 → 批量建剩余用例
```

**教训**: 审计能力应该在第一条用例创建前就准备好，用它来验证设计是否正确。
把审计当作 CI 的 lint，不是上线后的 hotfix。

## 二、细节问题（API & 代码实现）

### 2.1 平台 API 的 method 字段是整数，不是字符串

**现象**: `create_step(method="POST")` 报错
`Cannot deserialize value of type java.lang.Integer from String "POST"`

**原因**: 平台 `method` 字段编码为 `0=GET, 1=POST, 2=PUT, 3=DELETE`，
是 Java Integer 不是 String。从 66888 的步骤详情 `"method": 1` 可以确认。

**修复**: `platform_client.py` 的 `create_step` 默认参数从 `method="POST"` 改为 `method=1`。

**举一反三**: 平台所有 Java 后端的枚举字段（protocol、type、status）都是整数，
接口文档不存在时，从 GET 响应的 raw data 反推类型是最可靠的方式。

### 2.2 平台不支持 PUT，更新必须走"GET → 改 → POST"

**现象**: `update_step` 用 `requests.put(url)` 报 `Request method 'PUT' not supported`。
改成 `requests.post(url="/flow/{id}")` 后仍然报 `Request method 'POST' not supported`。

**原因**: 平台所有实体的更新都走同一个模式——
先 `GET /flow/{id}` 拿到完整对象，修改需要的字段，再 `POST /flow`（集合端点，非 /{id}）。
`update_case` 早已正确实现了这个模式，但 `update_step` 是后来新增的，
抄了一个常规 RESTful 的 PUT 写法，没有对齐已有实现。

**修复**: `update_step` 重写为 GET-modify-POST 模式，与 `update_case` 一致。

**教训**: 对同一个平台的不同资源（case、flow、variable），
更新模式应该保持一致。新增方法时优先参考已有的同平台方法实现方式。

### 2.3 body 字段的双重 JSON 序列化

**现象**: `update_step(body=json.dumps(body_dict))` 导致平台收到的 body 变成
`"{\\"key\\":\\"value\\"}"` 而非 `"{\"key\":\"value\"}"`。

**原因**: `platform_client.update_step` 内部已经对 body 做了 `json.dumps()`，
调用方再传 `json.dumps()` 的结果进来就变成了双重序列化。

**修复**: 调用方直接传 dict，让 `update_step` 内部统一做序列化。

**教训**: 和序列化相关的职责要归到唯一一层。要么调用方序列化，
要么被调用方序列化，绝不能两边都做。

### 2.4 审计 step_state_mapping 的 int/str 类型不匹配

**现象**: 审计结果显示 80/80 全部 CONFIG_MISMATCH——
`配置步骤「xxx」(ID:66888)不在已知状态映射中，无法校验`。

**原因**: `config_rules.step_state_mapping` 的 key 是字符串 `"66888"`，
但 `step.quote_id` 已被 `int()` 转为整数 `66888`。
`mapping.get(66888)` 查不到字符串 key `"66888"`。

**修复**: `_get_config_step_state` 中改为 `mapping.get(str(step.quote_id))`。

**教训**: Python 的 dict key 区分 int 和 str。
凡是从外部（JSON、API）读入的 ID，在做 dict lookup 时要统一类型。

### 2.5 首次审计的 77/80 误报——全局公共用例 ID 不在本地知识库

**现象**: 第一轮审计报 77/80 偏差，大量标记为 COMMON_MISSING。

**原因**: 51401、51403 等全局公共用例不在 `common_dir_id=66863` 目录下，
debugger 只扫描了指定目录的公共用例，认不出全局的。

**修复**: 对 debugger 中识别不了的 quoteId，增加平台 API 反查
（`GET /case/{id}` 验证是否存在），存在则不算缺失。

**教训**: 公共用例有两类——业务域公共用例（在指定目录下）和全局公共用例（平台级共享）。
审计工具要兼顾两者，不能只看本地目录。

### 2.6 已有公共步骤的 body 不是"标题说的那样"

**现象**: 66889 叫"关闭首次登录"，但查其步骤 body 发现 onex 相关字段全部为 0——
不只是关了首次，弱密码和定时也全关了。66890 同理。

**原因**: 这些公共步骤是手工在前端创建时随意填的，没有严格对照 body 模板。

**修复**: 以唯一正确的 66888（全开）的 body 为模板，按排列组合派生全部 8 种 body。

**教训**: 公共步骤的 body 必须从一个已验证的"全量模板"派生，
不能信任手工创建的步骤内容——哪怕名称看起来对，body 里可能完全不对。
检查 body 实际内容比看名称重要得多。

## 三、固化后的标准流程

经过此次重构，用例自动化搭建的正确流程应该是：

```
┌──────────────────────────────────────────────────────────────┐
│  Phase 0: 手工用例深度分析                                      │
│  ┌─────────────────────────────────────────────────────────┐ │
│  │ 1. 拆解标题/前置/操作/预期 → 结构化维度                     │ │
│  │ 2. 枚举所有独立维度的排列组合 → 状态组合表                   │ │
│  │ 3. 每种组合对应的 API 参数 → 抓包或查文档确认               │ │
│  │ 4. 不确定的参数 → 标记为待人工确认，去前端操作抓报文         │ │
│  │ 5. 输出: 「状态组合 × 公共步骤映射表 × body模板」          │ │
│  └─────────────────────────────────────────────────────────┘ │
│                            ↓                                  │
│  Phase 1: 公共步骤全量创建                                      │
│  ┌─────────────────────────────────────────────────────────┐ │
│  │ 1. 从已验证的 body 模板派生全部组合                         │ │
│  │ 2. 单条 API 调用验证 → 确认字段类型和格式                   │ │
│  │ 3. 批量创建所有公共步骤                                     │ │
│  │ 4. 更新知识库 manifest                                     │ │
│  └─────────────────────────────────────────────────────────┘ │
│                            ↓                                  │
│  Phase 2: 样本验证                                             │
│  ┌─────────────────────────────────────────────────────────┐ │
│  │ 1. 每种组合创建 1 条样本业务用例                            │ │
│  │ 2. 用 case-debugger 审计这 N 条样本 → 0 偏差               │ │
│  │ 3. 不通过 → 回 Phase 0 修正                               │ │
│  └─────────────────────────────────────────────────────────┘ │
│                            ↓                                  │
│  Phase 3: 批量创建 + 全量审计                                   │
│  ┌─────────────────────────────────────────────────────────┐ │
│  │ 1. 批量创建剩余业务用例                                     │ │
│  │ 2. 全量审计 → CONFIG_MISMATCH = 0                         │ │
│  │ 3. 同步到飞书、更新知识库                                   │ │
│  └─────────────────────────────────────────────────────────┘ │
└──────────────────────────────────────────────────────────────┘
```

---

## 四、一句话总结

> **先穷举状态组合，再创建公共步骤，最后批量生成业务用例——顺序反了，成本翻倍。**
